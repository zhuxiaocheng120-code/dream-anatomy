# Unify Analysis And Dream Result Card Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make quick decode and guided analysis return, render, and save analysis text plus Dream Result Card from one final analysis context.

**Architecture:** Keep the existing Express `/api/dream-analysis` backend proxy and existing `dream_records.report_content` storage. Add combined response shapes for `quick`, `guided_questions`, and `guided_final`, reuse `DreamResultCard.normalizeDreamResultCard()`, and render saved cards directly in current result pages. Dream Journal and Dream Detail become read-only consumers of saved `reportContent.dreamResultCard`, except old records can still show a manual generation fallback.

**Tech Stack:** Plain HTML/CSS/JavaScript, Node.js/Express, native fetch, Supabase JS through existing client, Node test runner.

## Global Constraints

- Keep product positioning as dream recording and self-exploration, not fortune telling, diagnosis, treatment, or future prediction.
- Use “可能”“也许”“可以理解为”“你可以思考”等 non-absolute language.
- DeepSeek API key must remain backend-only through `process.env.DEEPSEEK_API_KEY`.
- Do not add database schema, backend tables, login changes, payment, membership, Timeline, Calendar, Favorites, Trash, or WeChat.
- Do not use `innerHTML` for user or AI content.
- Do not log complete dream text, complete AI output, tokens, sessions, or keys.
- Quick final analysis returns `analysis` and `dreamResultCard` in one request.
- Guided final analysis returns deep report and `dreamResultCard` in one request after dynamic questions and user answers.
- Dream Journal and Dream Detail must read saved result data and must not automatically repeat AI calls.
- Missing Dream Result Card data must not display false 0 scores as if AI generated them.

---

## Root Cause Findings

1. Quick decode currently calls `/api/dream-analysis` once with `analysisType: "quick"` for text only, while Dream Result Card requires a second `analysisType: "result_card"` request from Dream Detail.
2. Quick result UI has no Dream Result Card mount, so users cannot see the card on the current result page after analysis.
3. Guided questions are generated by local `generateGuidedQuestions()` and are mostly fixed templates with only the first question including dream text.
4. Guided final report is generated by local `generateMockDeepReport()` and does not call DeepSeek.
5. `normalizeDreamResultCard()` turns missing scores into `0`, which makes incomplete cards look like real zero-valued AI results.
6. `createDreamRecord()` and `createDeepDreamRecord()` save text reports to `reportContent` but do not save a card generated in the same final analysis context.
7. Dream Detail can request a card for old records when the user clicks the fallback button; this is allowed for old records, but new quick/guided records should already have saved cards.

## File Structure

- Modify `server.js`: add request validation and prompts for `guided_questions`, `guided_final`, combined quick output, strict combined normalization, and safe partial response when analysis succeeds but card fails.
- Modify `src/dreamResultCard.js`: distinguish generated, failed, unavailable, fallback, and legacy missing states; avoid false zero scores for missing dimensions.
- Modify `src/index.html`: add quick/guided Dream Result Card mount containers and update mock copy.
- Modify `src/app.js`: request combined quick/deep responses, render cards on current pages, save `reportContent.dreamResultCard`, and keep journal/detail read-only for saved data.
- Modify `src/style.css`: style result-card mounts inside current result pages without changing broader UI.
- Modify `README.md` and `docs/PROJECT_STATUS.md`: update current state and boundaries.
- Modify `tests/server.test.js`: cover combined response shapes, guided dynamic questions, partial card failure, and unsupported types.
- Modify `tests/dreamResultCard.test.js`: cover missing score unavailable state and failed card copy.
- Modify `tests/dreamJournal.test.js`: cover quick/guided current-page cards, saved report content, no repeated AI calls on Journal/Detail.
- Modify `tests/dreamSync.test.js` only if report content mapping needs additional proof.
